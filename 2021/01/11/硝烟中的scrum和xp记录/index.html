硝烟中的scrum和xp记录

产品backlog(story):
  • ID 统一标识符
  • Name 名称一般2-10个字
  • Importance 重要性 一般10-150 分数越高越重要
  • Initial estimate 初步估算 最小的单位是story point，一般是“理想的人-天”
  • How to demo 描述这个事情如何在sprint上示范 如果使用的TDD，那么这段时间的描述可以作为验收测试的伪码表示
  • Notes 注解 相关的信息、解释说明和对其他资料的引用

事例：

ID Name Importance es How to demo Notes
1 存款 30 5 登陆，打开存款界面，存入10块钱，余额就多了10块钱 目前不考虑加密
2 查看交易明细 10 8 登陆，点击“交易”，存入一笔款项，返回交易页面，看到记录 使用分页，避免大规模查询数据库

其他的backlog字段

  • Track 类别
  • Component 组件
  • Requestor 请求者
  • Bug tracking ID bug追踪id
准备sprint计划
  • 产品backlog必须存在
  • 一个产品只能有一个产品backlog和一个产品负责人
  • 所有重要的backlog条目都根据重要性评过分
  • 产品负责人应当理解每个story的含义

产品负责人之外的人可以向backlog添加story但是不能评估Importance和时间，这分别是产品负责人和开发团队的权利

通过计划会议制定sprint计划
  • sprint目标
  • 团队成员名单（以及投入程度，如果不是100%的话）
  • sprint backlog
  • 确定sprint演示日期
  • 确定时间地点举行每日scrum会议
sprint优先级列表（从高到低）
  • sprint演示日期和目标
  • 经过团队认可、添加到这个sprint中的故事列表
  • sprint每个故事的估算值
  • sprint每个故事的“how to demo”
  • 生产率和资源计算，用作sprint计算的现实核查
  • 明确每日例会固定举行的时间地点
  • 把故事拆分成任务
